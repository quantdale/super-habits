## 1. Project Identity

- **What it does**: SuperHabits is an offline-first, cross-platform personal productivity app that consolidates five modules—to-do lists, daily habit tracking with progress rings, a 25-minute Pomodoro timer, workout routine logging, and manual calorie/nutrition tracking—into a single tabbed interface. It targets individual users who want a unified self-improvement dashboard that works without network connectivity. Data is persisted locally in SQLite and the architecture is pre-wired for future cloud sync via Supabase.
- **Current status**: Active development / MVP. The sync engine is a no-op stub, Supabase integration is deferred, and the calorie screen hard-codes meal type to "snack." No production deployment pipeline exists.
- **Tech stack**:
  - Language: TypeScript ~5.9.2
  - Runtime: React 19.2.0, React Native 0.83.2
  - Framework: Expo ~55.0.5 with expo-router ~55.0.4 (file-based routing)
  - Styling: NativeWind ^4.2.2 (Tailwind CSS ^3.4.19 for React Native)
  - Database: expo-sqlite ^55.0.10 (SQLite WAL mode, local-first)
  - State/Cache: @tanstack/react-query ^5.90.21, zustand ^5.0.11 (zustand imported but unused in source currently)
  - List rendering: @shopify/flash-list 2.0.2
  - Animations: react-native-reanimated 4.2.1
  - SVG: react-native-svg 15.15.3

  - ID generation: custom prefix+timestamp+random (lib/id.ts)
  - Date handling: date-fns ^4.1.0 (dependency listed, not directly imported in source—time utilities use native Date)
  - PWA: workbox-window ^7.4.0, custom service worker
  - Bundler: Metro (via Expo)
  - Test runner: Vitest ^3.2.4
- **Entry points**:

## 3. Architecture

### 3.1 High-Level Architecture

The app follows a **feature-module** architecture with three layers:

```
┌─────────────────────────────────────────────┐
│                 app/ (Routing)               │
│  expo-router file-based routes → tab layout  │
│  Each route delegates to a feature Screen    │
├─────────────────────────────────────────────┤
│              features/ (Modules)             │
│  *.Screen.tsx  — UI (React components)       │
│  *.data.ts     — Data access (SQLite CRUD)   │
│  *.domain.ts   — Pure business logic         │
├─────────────────────────────────────────────┤
│              core/ (Infrastructure)           │
│  db/         — SQLite client + types         │
│  ui/         — Shared presentational atoms   │
│  sync/       — Sync engine (noop stub)       │
│  auth/       — Guest profile management      │
│  providers/  — React context providers       │
│  pwa/        — Service worker registration   │
├─────────────────────────────────────────────┤
│              lib/ (Shared Utilities)          │
│  id, time, notifications, supabase stub      │
└─────────────────────────────────────────────┘
```

Communication is exclusively via **direct function calls** (async, Promise-based). There are no HTTP calls, no event buses, no message queues. The sync engine has a queue but it is never flushed in the current codebase.

### 3.2 Data Flow

**Primary flow (Todo creation as representative example):**

```
User taps "Add task" in TodosScreen
  → TodosScreen.onCreate() (features/todos/TodosScreen.tsx:32)
    → addTodo({ title, notes }) (features/todos/todos.data.ts:14)
      → createId("todo") (lib/id.ts:1) — generates unique ID
      → nowIso() (lib/time.ts:1) — generates ISO timestamp
      → getDatabase() (core/db/client.ts:120) — returns singleton SQLite handle
      → db.runAsync(INSERT INTO todos ...) — persists to SQLite
      → syncEngine.enqueue({ entity, id, ... }) (core/sync/sync.engine.ts:28) — queues sync record (noop)
    → refresh() → listTodos() → db.getAllAsync(SELECT ...) → setItems(results)
      → React re-renders the FlashList
```

**Habit increment flow (secondary):**

```
User taps HabitCircle
  → HabitsScreen.handleIncrement(habitId) (features/habits/HabitsScreen.tsx:133)
    → incrementHabit(habitId) (features/habits/habits.data.ts:34)
```

**Pomodoro timer flow (async/background):**

```
User taps "Start focus"
  → PomodoroScreen.start() (features/pomodoro/PomodoroScreen.tsx:48)
    → scheduleTimerEndNotification(FOCUS_SECONDS, ...) (lib/notifications.ts:30) — schedules OS notification
    → setIsRunning(true) → useEffect starts setInterval (PomodoroScreen.tsx:25)
      → Every 1s: setRemaining(prev - 1)
      → At remaining === 0:
        → logPomodoroSession(...) (features/pomodoro/pomodoro.data.ts:6) — persists to SQLite
        → setHistoryVersion(v + 1) → triggers history reload useEffect
```

### 3.3 State Management

- **Persistent state**: SQLite database `superhabits.db` — all user data (todos, habits, habit_completions, pomodoro_sessions, workout_routines, workout_logs, calorie_entries, app_meta). WAL journal mode.
- **UI state**: React `useState` hooks local to each Screen component. No global state store is actively used (zustand is a dependency but no store file exists).
- **Server cache**: `@tanstack/react-query` QueryClient is instantiated in `AppProviders.tsx:8` but no `useQuery`/`useMutation` hooks are used anywhere. All data fetching is done with direct async calls + local state.
- **Sync queue**: In-memory array in `SyncEngine` (core/sync/sync.engine.ts:26). Records are enqueued but never flushed. Lost on app restart.
- **Guest profile**: Stored in `app_meta` table as JSON under key `guest_profile`.
- **Mutable state**: All SQLite data is mutable. All React component state is mutable via setState.
- **Consistency guarantees**: SQLite WAL mode provides serializable writes. No multi-process concurrency concerns (single Expo process). `UNIQUE(habit_id, date_key)` constraint prevents duplicate daily completions.

### 3.4 Concurrency Model

- **Single-threaded JavaScript event loop** (React Native / Hermes on native, browser main thread on web).
- **Timer**: Pomodoro uses `setInterval` at 1-second intervals (PomodoroScreen.tsx:26). The closure captures `startedAt` from component state. The effect cleanup clears the interval.
- **Async I/O**: All database operations are `async`/`await` via expo-sqlite's async API. They run on a native thread and return Promises to JS.
- **Race condition risks**:
  - Pomodoro timer effect depends on `startedAt` in its closure. If `start()` is called rapidly multiple times, multiple intervals could run simultaneously because `setIsRunning(true)` triggers the effect, but the `isRunning` check in the effect dependency array could race. Mitigated partially by the `if (!isRunning) return` guard at line 25.
  - Habit increment/decrement: The read-then-write pattern in `incrementHabit`/`decrementHabit` (habits.data.ts:34-72) is not wrapped in a transaction. If two rapid taps fire before the first write completes, the count could be incremented once instead of twice. Low risk because of single-threaded JS + sequential await.

### 3.5 External Dependencies

- **APIs called**: None. The app is fully offline. `lib/supabase.ts` has a `remoteMode` flag set to `"disabled"` with a comment "Post-MVP: initialize Supabase client."
- **Databases**: Local SQLite file `superhabits.db` via expo-sqlite. On web, this uses OPFS (Origin Private File System) backed by WASM.
- **External services**: expo-notifications (OS-level notification scheduling, no remote push). No other external services.
- **File system dependencies**: SQLite DB file managed by expo-sqlite (path is platform-dependent). Service worker caches `/` and `/index.html` on web.

## 7. Conventions & Patterns

### 7.1 Naming Conventions

- **Files**: camelCase for source (`guestProfile.ts`, `habits.data.ts`), PascalCase for React components (`HabitCircle.tsx`, `CaloriesScreen.tsx`)
- **File suffixes**: `*.data.ts` = data access layer, `*.domain.ts` = pure business logic, `*Screen.tsx` = screen-level UI component
- **Directories**: lowercase (`core/`, `features/`, `lib/`, `app/`)
- **Functions**: camelCase (`listTodos`, `addHabit`, `calculateHabitProgress`)
- **Types**: PascalCase (`Todo`, `Habit`, `CalorieEntry`, `BaseEntity`)
- **Constants**: UPPER_SNAKE_CASE (`FOCUS_SECONDS`, `HABIT_ICONS`, `DEFAULT_HABIT_COLOR`, `NAV_ITEMS`)
- **Database columns**: snake_case (`target_per_day`, `created_at`, `deleted_at`)
- **IDs**: prefixed format `{entity}_{timestamp}_{random}` (lib/id.ts:1)
- **Routes**: kebab-case tab names matching feature names (`todos`, `habits`, `pomodoro`, `workout`, `calories`)

### 7.2 Code Patterns

- **Feature module pattern**: Each feature has up to three files: `*.data.ts` (SQLite operations), `*.domain.ts` (pure logic), `*Screen.tsx` (UI). Not all features have domain files.
- **Soft delete pattern**: All mutable entities use `deleted_at TEXT` column. Delete = `UPDATE SET deleted_at = now()`. List queries filter `WHERE deleted_at IS NULL`.
- **Data refresh pattern**: Each screen has a `refresh()` callback that re-fetches from SQLite and calls setState. `useFocusEffect` triggers refresh on tab focus. Mutations call `refresh()` after await.
- **Error handling**: `.catch(() => undefined)` for non-critical initialization (AppProviders.tsx:12,14). `Alert.alert()` for user-facing validation errors. Silent catch for migration ALTER TABLE (client.ts:87,97-103). No global error boundary.
- **Sync enqueue pattern**: Every write operation in `*.data.ts` calls `syncEngine.enqueue(...)` with the operation type, but this is currently a no-op.
- **No logging**: No console.log, no logger. No telemetry.
- **Comment conventions**: Minimal comments. Inline comments only for non-obvious behavior (e.g., migration catch blocks). No JSDoc.

### 7.3 Data Conventions

- **IDs**: Text, format `{prefix}_{timestamp}_{random8}` — e.g., `"todo_1710507600000_a1b2c3d4"`. Generated by `lib/id.ts`. Guest profile uses `"guest_{timestamp}"`.
- **Timestamps**: ISO 8601 strings (`new Date().toISOString()`), stored as TEXT in SQLite. Always UTC.
- **Date keys**: `YYYY-MM-DD` format (e.g., `"2026-03-15"`). Used for habit_completions.date_key and calorie_entries.consumed_on. Generated by `lib/time.ts:toDateKey()` which slices the local ISO string, making it **local timezone dependent**.
- **Booleans in SQLite**: Stored as INTEGER 0/1 (e.g., `todos.completed`). TypeScript type uses `0 | 1` literal union.
- **Colors**: Hex strings (`"#64748b"`). Stored as TEXT in SQLite.
- **Enum-like values**: Stored as TEXT strings in SQLite. Validated only by TypeScript union types at compile time, not at the database level.

## 8. Architectural Invariants — Rules That Must Never Be Broken

### Inferred Invariant 1: Database singleton must be initialized before any data access
- **What**: `getDatabase()` must be called (and its Promise resolved) before any `*.data.ts` function executes a query.
- **Why**: All data functions call `await getDatabase()` which returns the singleton Promise. If the bootstrap DDL has not run, tables won't exist and queries will fail with "no such table."
- **Where enforced**: `core/db/client.ts:120-125` (lazy singleton pattern), `core/providers/AppProviders.tsx:12` (eagerly calls `initializeDatabase()` on mount).
- **How an agent could accidentally violate it**: Calling a data function at module scope or in a top-level import side effect before AppProviders has mounted.

### Inferred Invariant 2: Soft deletes must use deleted_at, never hard DELETE for main entities
- **What**: Todos, habits, workout_routines, and calorie_entries must be soft-deleted by setting `deleted_at` to an ISO timestamp. Hard SQL `DELETE` is reserved only for habit_completions (decrementHabit, habits.data.ts:64).
- **Why**: The sync engine queues "delete" operations referencing entity IDs. If rows are hard-deleted, future sync (when implemented) would not find them to reconcile. All list queries filter `WHERE deleted_at IS NULL`.
- **Where enforced**: `removeTodo` (todos.data.ts:33), `deleteHabit` (habits.data.ts:110), `deleteRoutine` (workout.data.ts:43), `deleteCalorieEntry` (calories.data.ts:46).
- **How an agent could accidentally violate it**: Using `DELETE FROM todos WHERE id = ?` instead of `UPDATE todos SET deleted_at = ?`.

### Inferred Invariant 3: Every write must enqueue a sync record
- **What**: Every create/update/delete operation on a synced entity must call `syncEngine.enqueue(...)`.
- **Why**: When cloud sync is implemented, the sync queue is the mechanism to push local changes. Missing enqueue calls would cause data loss on sync.
- **Where enforced**: All `*.data.ts` files (todos.data.ts:22,30,37; habits.data.ts:31,107,114; workout.data.ts:22,51; calories.data.ts:43,50).
- **Exception**: `pomodoro.data.ts` does NOT enqueue sync records. `completeRoutine` in workout.data.ts does NOT enqueue for workout_logs (only for workout_routines). `habit_completions` increment/decrement do NOT enqueue.
- **How an agent could accidentally violate it**: Adding a new write function without the syncEngine.enqueue call.

### Inferred Invariant 4: IDs must be generated with createId(prefix)
- **What**: All entity IDs must use the `createId(prefix)` function from `lib/id.ts`.
- **Why**: The prefix encodes entity type, which is useful for debugging and future sync disambiguation. The timestamp component provides rough ordering. Using `uuid` or random strings would break this convention.
- **Where enforced**: Every `add*`/`log*`/`increment*` function in `*.data.ts` files.
- **How an agent could accidentally violate it**: Using `uuid.v4()` or `Math.random()` directly instead of `createId()`.

### Inferred Invariant 5: Date keys must use local timezone YYYY-MM-DD
- **What**: The `toDateKey()` function generates date keys by slicing the ISO string of a local Date object. This means date boundaries are local-timezone-dependent.
- **Why**: If UTC date keys were used, a user completing a habit at 11 PM local time might see it recorded under tomorrow's date. The local-timezone slicing is deliberate.
- **Where enforced**: `lib/time.ts:5-7` — `date.toISOString().slice(0, 10)`.
- **How an agent could accidentally violate it**: Using `new Date().toISOString().slice(0, 10)` directly (this would produce UTC dates) or changing `toDateKey()` to use UTC.
- **IMPORTANT NOTE**: `toISOString()` actually returns UTC, so the current implementation of `toDateKey()` **does** produce UTC date keys, not local timezone keys. This is a latent bug — a user at UTC-5 completing a habit at 11 PM local time (04:00 UTC next day) would get tomorrow's date key.

### Inferred Invariant 6: habit_completions UNIQUE(habit_id, date_key) constraint
- **What**: The database enforces at most one row per habit per date in the `habit_completions` table.
- **Why**: The increment/decrement logic does a SELECT then UPDATE on this row. If duplicates existed, the SELECT would return an arbitrary row and counts would be inconsistent.
- **Where enforced**: DDL in `core/db/client.ts:35` — `UNIQUE(habit_id, date_key)`.
- **How an agent could accidentally violate it**: Inserting into habit_completions without checking the UNIQUE constraint, or removing the constraint in a migration.

### Inferred Invariant 7: Schema version must be monotonically increasing
- **What**: The `app_meta` key `db_schema_version` must only increase. Migrations check `if (version < N)` and run forward-only alterations.
- **Why**: There are no down-migrations. Running a migration out of order or decrementing the version would re-run ALTER TABLE statements (which would silently fail due to try/catch, but the inconsistency could cause issues).
- **Where enforced**: `core/db/client.ts:79-108` — `runMigrations()`.
- **How an agent could accidentally violate it**: Setting `db_schema_version` to a lower value, or adding a migration with a number less than the current max (3).

### Known bugs/limitations

1. **Date key timezone bug**: `toDateKey()` in `lib/time.ts:6` uses `date.toISOString().slice(0, 10)`. `toISOString()` returns UTC time, so for users in negative UTC offsets, activities done late evening will be recorded under the next day's date.
2. **Meal type hard-coded**: `CaloriesScreen.tsx:55` always passes `mealType: "snack"`. The `CalorieEntry` type supports "breakfast" | "lunch" | "dinner" | "snack" but there is no selector in the UI.
3. **nextPomodoroState unused**: `pomodoro.domain.ts` exports `nextPomodoroState` but `PomodoroScreen.tsx` does not use it — it has inline state logic instead.
4. **Pomodoro sessions not synced**: `pomodoro.data.ts` does not call `syncEngine.enqueue()`, unlike all other data modules.
5. **Workout logs not synced**: `completeRoutine()` in `workout.data.ts:25` does not enqueue a sync record for workout_logs.
6. **Habit completion records not synced**: `incrementHabit`/`decrementHabit` do not enqueue sync records.
7. **schema.sql out of date**: `core/db/schema.sql` does not include `category`, `icon`, `color` columns on the habits table (these were added in migrations 2 and 3).
8. **Sync queue never flushed**: `SyncEngine.flush()` is never called anywhere. Records accumulate in memory forever until app restart.
9. **zustand unused**: `zustand` is listed as a dependency but no store is defined anywhere.
10. **date-fns unused**: `date-fns` is listed as a dependency but no imports of it exist in the source code.
11. **uuid unused**: `uuid` is listed as a dependency but `lib/id.ts` uses custom `Math.random().toString(36)` instead.
12. **expo-background-fetch and expo-task-manager unused**: Listed as dependencies but not imported anywhere.
13. **No error boundary**: No React error boundary exists. An unhandled error in any screen crashes the entire app.
14. **QueryClient unused**: `@tanstack/react-query` QueryClient is created and provided but no component uses `useQuery` or `useMutation`.
15. **App.tsx / index.ts are dead code**: Not used by expo-router.

## 13. Security Considerations

- **Authentication**: Guest-only. No user authentication. Guest profile ID is `guest_{timestamp}` — not cryptographically random but sufficient for local-only use.
- **Input validation**: Minimal. Title/name presence is checked via `Alert.alert()` in Screen components. Calorie value must be a positive finite number (CaloriesScreen.tsx:43). No SQL injection risk because all queries use parameterized statements (`?` placeholders).
- **Sensitive data**: No PII beyond user-entered todo text and food names. No passwords, tokens, or API keys.
- **Known security gaps**:
  - The `createId` function (lib/id.ts:2) uses `Math.random()` which is not cryptographically secure. For local IDs this is acceptable, but if IDs are ever exposed to a server, they should use `crypto.getRandomValues()` or the `uuid` package.
  - Service worker (public/sw.js) caches all GET responses including potential sensitive data. No cache size limit or TTL.
  - COOP/COEP headers are set for SharedArrayBuffer support but also affect cross-origin embedding behavior.
- **CSP/CORS**: No Content-Security-Policy configured. COOP/COEP configured for expo-sqlite WASM support.

## 14. Performance Characteristics

- **Hot paths**:
  - Tab focus → `refresh()` → SQLite SELECT → setState → re-render. This runs every time the user switches tabs (`useFocusEffect`).
  - Habits screen refresh: `listHabits()` followed by N individual `getHabitCountByDate()` calls (one per habit) — this is an **N+1 query pattern** (HabitsScreen.tsx:74).
  - Pomodoro timer: `setInterval` fires every 1 second, calling `setRemaining()` which triggers a re-render each second.
- **Known bottlenecks**:
  - **Habits N+1**: For H habits, the habits screen executes 1 + H queries on every refresh. With many habits, this will degrade. Could be solved with a JOIN query.
  - **Full table scans**: No indexes exist beyond PRIMARY KEY. Queries like `SELECT * FROM calorie_entries WHERE ... AND consumed_on = ?` do full table scans on `consumed_on`.
- **Caching**: No application-level caching. React Query is provided but unused. Every tab focus re-queries SQLite.
- **Memory usage**: Sync queue grows unbounded (core/sync/sync.engine.ts:26). With heavy use, the in-memory queue array will grow indefinitely until app restart.
- **List rendering**: TodosScreen uses FlashList (virtualized). Other lists (habits, pomodoro history, workout logs, calorie entries) use `.map()` rendering inside ScrollView — not virtualized, all items rendered.

## 15. Glossary

| Term | Meaning |
|------|---------|
| **date_key** | A `YYYY-MM-DD` string identifying a calendar day. Used to partition habit completions and calorie entries by day. |
| **soft delete** | Setting `deleted_at` to a timestamp instead of removing the row. The row is excluded from queries via `WHERE deleted_at IS NULL`. |
| **BaseEntity** | TypeScript type providing the standard fields `id`, `created_at`, `updated_at`, `deleted_at` shared by most entities. |
| **target_per_day** | The goal number of times a habit should be completed per day. Progress = count / target_per_day, clamped to [0, 1]. |
| **habit_completion** | A row in `habit_completions` tracking how many times a habit was performed on a specific date. One row per habit per day (UNIQUE constraint). |
| **HabitCategory** | Time-of-day grouping for habits: "anytime", "morning", "afternoon", "evening". Determines display grouping on HabitsScreen. |
| **HabitIcon** | One of 12 MaterialIcons names used as visual identifiers for habits. |
| **FOCUS_SECONDS** | Constant 1500 (25 minutes) — the Pomodoro focus duration. Defined in PomodoroScreen.tsx:11. |
| **SyncRecord** | A record of a local data change (entity, id, timestamp, operation type) queued for future cloud sync. |
| **SyncAdapter** | Interface for push/pull sync with a remote server. Only NoopSyncAdapter is implemented. |
| **guest_profile** | An `app_meta` entry storing a JSON-serialized `{ id, createdAt }` object. Created on first app launch. |
| **app_meta** | A key-value table storing application metadata: guest profile JSON, database schema version. |
| **WAL** | Write-Ahead Logging — SQLite journal mode enabling concurrent reads during writes. Set in db bootstrap (client.ts:6). |
| **brand-500** | The primary brand color `#4f79ff` (blue), defined in tailwind.config.js:9 and used throughout the UI. |
| **NativeWind** | Library that brings Tailwind CSS to React Native via the `className` prop. Requires babel and metro configuration. |
| **COOP/COEP** | Cross-Origin-Opener-Policy / Cross-Origin-Embedder-Policy headers. Required for SharedArrayBuffer, which expo-sqlite uses for web WASM support. |
| **patch-package** | Tool that applies diff patches to node_modules after install. Two patches exist for Windows compatibility and dev tool fixes. |

## 16. Quick Reference for Agents

### Before writing any code
- [ ] Read `core/db/client.ts` — understand the database schema, bootstrap process, and migration pattern
- [ ] Read `core/db/types.ts` — understand all entity types and their field shapes
- [ ] Read the relevant `features/{module}/{module}.data.ts` file for the module you're changing
- [ ] Read `lib/id.ts` and `lib/time.ts` — understand ID format and timestamp conventions
- [ ] Understand the soft-delete pattern: never use `DELETE FROM` on main entities; always set `deleted_at`

### The most dangerous things to get wrong
1. **Hard-deleting rows** instead of soft-deleting (setting `deleted_at`) — will break future sync and violate the universal pattern
2. **Forgetting `syncEngine.enqueue()`** after a write operation — will cause data loss when sync is implemented
3. **Using raw SQL `DELETE` on main entities** — see #1
4. **Adding a new SQLite column without a migration** in `client.ts:runMigrations()` — existing users' databases won't have the column, causing runtime crashes
5. **Breaking the database singleton** by creating a second `openDatabaseAsync()` call — would cause two connections with potentially inconsistent schema states

| Task | Start here |
|------|-----------|
| Add a new feature module | Copy pattern from `features/todos/` (data + screen), add route in `app/(tabs)/`, add tab in `app/(tabs)/_layout.tsx` |
