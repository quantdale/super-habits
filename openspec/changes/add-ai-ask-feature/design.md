## Context

`features/command/` today is a one-shot, write-only pipeline: `CommandScreen` collects free text, `commandParser` (facade over `mockCommandParser` and `realCommandParser`) turns it into a `create_todo`/`create_habit` draft, the user reviews/edits the draft, and `command.executor` writes it on confirm. The remote path (`realCommandParser.ts`) calls the Supabase edge function `parse-ai-command`, which sends the model only `{rawText, nowIso, locale, timeZone, todayDateKey, tomorrowDateKey}` — no stored user data ever leaves the device today.

This change adds a second, parallel capability — Ask — that answers natural-language questions about the user's own locally-stored data (pending todos, calorie totals, habit streaks) with no write. Because SQLite is the on-device source of truth and there is no server-side copy of it, retrieval cannot happen in the edge function; it has to happen client-side, against the same `.data.ts`/`.domain.ts` read functions the screens already use, before anything is sent over the network.

`single-page-consolidation` is a separate, already-shipped change to command's routing/wiring (deriving launch context from `NavigationContext`, removing the standalone `/command` route). This design does not touch or depend on it.

## Goals / Non-Goals

**Goals:**
- Answer v1 questions about pending todos, calorie summaries (today + range), and habit streaks (per-habit and overall), entirely from local data.
- Guarantee the model never computes a number from raw rows — all arithmetic happens in existing/new TypeScript `.domain.ts` functions; the model only classifies intent and phrases the final answer.
- Reuse existing patterns wherever one already fits: the `CommandCenterContext` state-management shape, the `commandInternalRollout.ts` AsyncStorage precedent, and the existing Supabase client-to-edge-function auth/invocation helpers. The model-call mechanism itself is a deliberate exception to "reuse `parse-ai-command`'s convention" — see Decisions below.
- Make the privacy shift (real user data leaving the device for the first time) visible in the proposal and in code structure, not buried.

**Non-Goals:**
- No agentic/multi-step tool-use — the model does not decide which local functions to call or loop through multiple tool invocations. A single forced tool-call per model invocation (used only as Bedrock/Claude's structured-output mechanism, functionally equivalent to OpenAI's `json_schema` response mode) is in scope and is how both the classify and phrase calls achieve structured output. The classify → local-dispatch → phrase shape stays a fixed two-call pattern decided entirely by client code, not something the model branches through itself.
- No vector search / embeddings. The retrievable surface is a small, fixed set of structured aggregate queries, not an open corpus.
- No disk-persisted or cross-session chat history. In-memory only, cleared on cold start.
- No new state management library (Zustand stays unused, per existing codebase convention).
- No workout, pomodoro, or linked-actions intents in v1.
- No keyword/regex pre-router in front of the classify call — every Ask question goes through the model, on the reasoning that a hand-maintained router would silently miss new phrasings as the intent set grows.
- No user-facing privacy disclosure UX — flagged as a required product/design follow-up, not resolved here.

## Decisions

**Two-call structured-output pipeline, not agentic tool-use.** Call 1 (`classify`): `{question, conversationContext?}` → model returns `{intent, params}` as structured output. Client code deterministically dispatches on `intent` to local data/domain calls. Call 2 (`phrase`): `{question, retrievedFacts}` → model returns `{answer: string}`, also structured output. Rejected alternative: a single call that receives raw rows and both classifies and answers — rejected because it reintroduces the exact hallucination/miscounting risk this design exists to avoid, and collapses the auditable boundary between "what the model decided to look up" and "what it said about it." (How structured output is actually obtained from the model differs by provider — see the Bedrock decision below; the two-call shape and the client-owned dispatch between them are provider-independent.)

**`user-ai-ask` calls AWS Bedrock (Claude Haiku), not OpenAI — a deliberate per-feature divergence, not an oversight.** `parse-ai-command` (Create) stays on OpenAI, completely unmodified. This is the user's own infrastructure choice for Ask specifically, not something inferred from Create's precedent, and the two functions are intentionally allowed to use different providers. Two mechanisms differ as a result, both scoped to `user-ai-ask` only:
- **Auth to the model provider.** OpenAI uses a bearer API key (`OPENAI_API_KEY`), as `parse-ai-command` does today. Bedrock uses AWS SigV4 request signing with IAM credentials (access key + secret, or an assumed role) — a materially different auth flow, not a drop-in secret swap. `user-ai-ask`'s request-signing code will look nothing like `parse-ai-command`'s bearer-token header. This is separate from — and must not be confused with — the client-to-edge-function auth leg (see below), which is unaffected.
- **Structured-output mechanism.** OpenAI's `response_format: { type: "json_schema", ... }` has no direct Bedrock/Claude equivalent. Structured output from Claude on Bedrock is obtained via Claude's tool-use mechanism: defining a single tool matching the desired output schema and forcing the model to call it (`tool_choice` forced to that one tool), then reading the tool-call input as the result. This is a single forced-tool-call per invocation, used purely as a structured-output mechanism — not an agentic loop (see the reworded Non-Goal above). Both the `classify` and `phrase` calls use this same forced-single-tool pattern.

**New edge function `user-ai-ask`, not an extension of `parse-ai-command`.** Keeps Create's request/response contract completely stable (zero risk of regressing the shipped create-draft flow). Two distinct auth legs, not to be conflated:
- **Client → edge function**: unaffected by the provider change. Reuses the same Supabase auth shape as `realCommandParser.ts` — `getSupabaseAccessToken()` for the bearer token, `getSupabaseAnonKey()` for the `apikey` header.
- **Edge function → model provider**: this is the leg that changes. `parse-ai-command` sends an OpenAI bearer key; `user-ai-ask` performs AWS SigV4 signing against Bedrock, per the decision above.
Duplicated boilerplate between the two functions is now smaller than a same-provider extension would have produced — CORS headers and request validation are still duplicated, but the model-invocation code is not shared boilerplate at all, since the two functions talk to entirely different providers with entirely different auth and structured-output mechanisms.

**Intent → retrieval mapping is a hand-written table, not a mega-query.** `pending_todos` → new `listPendingTodos()`/`countPendingTodos()` on `features/todos/todos.data.ts` (today, "pending" filtering only exists ad hoc inside `TodosScreen.tsx`/`OverviewScreen.tsx` component code — this promotes it to the data layer, matching this repo's layering rule that `*.data.ts` owns all reads). `calorie_summary` → `getCalorieSummaryByRange(startDateKey, endDateKey)` + `caloriesTotal()` (both already exist and are stable; "today" is just `startDateKey === endDateKey === toDateKey()`). `habit_streak` → `listHabits()` to resolve a name to a habit id, `getCompletionHistory(habitId, days)` + `buildDayCompletions(completions, targetPerDay, days)` + `calculateCurrentStreak`/`calculateLongestStreak` (all already exist, unchanged). A new thin orchestration module in `features/command/` owns this dispatch table — it is the only new "retrieval" code; everything it calls already exists except the two new todo functions.

**Mode toggle: Ask / Create / Auto, segmented control.** `Auto` runs the same `classify`-shaped call first to decide Ask vs. Create before routing into either existing (Create) or new (Ask) pipeline — it does not introduce a third pipeline. Last-used mode persists via AsyncStorage key `superhabits.command.last-used-mode`, read/written with the exact same cache-then-persist shape as `getAiCommandInternalRolloutPreference`/`setAiCommandInternalRolloutPreference` in `commandInternalRollout.ts`. Defaults to `"auto"` when absent (fresh install or existing users pre-upgrade).

**Chat history: new `AskConversationContext`, in-memory, session-scoped to the app process (not the modal).** Structurally parallel to `CommandCenterContext` (`createContext` + `useState`, no external store) — consistent with this codebase's stated preference for local React state over the installed-but-unused Zustand/React Query. "Session" here means the app's process lifetime, not whether the command-center modal is currently open: closing and reopening the modal SHALL NOT clear history — a user who dismisses the sheet mid-conversation and reopens it should still see prior turns. Only app cold start clears it (the context's state is initialized once at provider mount, not reset on modal close). Not read or written to SQLite or AsyncStorage at any point. Follow-up questions resend prior turns as `conversationContext` in the `classify` call. If persistence is wanted later, the closer-fitting precedent already in this codebase is the `app_meta` JSON-blob pattern (used for `pomodoro_settings`, `guest_profile`), not a new table — noted here as a pointer for a future change, not decided now.

**Auto-mode misroute recovery: single alternate-mode retry button.** When Auto routes an input to the wrong pipeline (a Create-shaped input classified as Ask, or vice versa), the result view shows whatever (possibly unhelpful) outcome that pipeline produced, plus one visible affordance — "Try as Create instead" / "Try as Ask instead" — that re-submits the exact same input text through the other pipeline without the user retyping anything. This is the actual v1 design, not a placeholder: no richer disambiguation UI (e.g. "did you mean" suggestions, confidence scores shown to the user) is in scope for v1. Rejected alternative: silently re-trying both pipelines and picking one — rejected because it hides the misroute from the user instead of giving them a one-tap correction, and doubles model calls on every Auto submission instead of only on a wrong guess.

**Privacy posture.** Unlike Create (sends only the raw command string), Ask's `phrase` call sends derived facts about real logged data (e.g., "3 entries totaling 1800 kcal today") to a third-party model API. This is called out explicitly in the proposal's Impact section as a product/design decision requiring its own disclosure, not something this design resolves through engineering alone.

## Risks / Trade-offs

- **Two model round-trips per question** → higher latency than Create's single call. Mitigation: both calls are schema-constrained (via forced single tool-call) and small (no raw rows in either payload), and Claude Haiku is a cost/latency-efficient tier by design. Latency-bucket observation (mirroring `CommandParseObservation`'s `fast`/`noticeable`/`frustrating` buckets) should be added to `user-ai-ask` calls so this is measured, not assumed.
- **Classify-call misses a phrasing → intent mismatch** → wrong or empty retrieval. Mitigation: classify response schema includes an explicit "unsupported/unclear" outcome (mirroring `ParseCommandResult`'s `unsupported` branch), surfaced to the user as "try rephrasing," not a silent wrong answer.
- **No pre-router means every question costs a network round-trip**, even ones a regex could answer for free (e.g., "pending todos"). Accepted trade-off per explicit decision — the alternative (a growing hand-maintained keyword router) was judged a worse long-term maintenance cost than the latency/cost of always calling the model.
- **Auto mode's classification is itself a third thing that can be wrong** (Ask vs. Create misroute). Mitigation: the visible single-retry-button recovery path described in Decisions above — not a silent failure.
- **Second provider/SDK dependency.** This change introduces AWS Bedrock alongside OpenAI (used by Create) as a second, structurally different AI provider integration — different auth (SigV4 vs. bearer key), different structured-output mechanism (forced tool-call vs. `json_schema`), likely a different SDK or raw HTTP-signing implementation in the Deno edge runtime. This is a deliberate choice, not a cost-free one: it is a real ongoing maintenance surface (two integrations to keep working, two sets of provider-specific failure modes to handle), not just a model-cost optimization. If a third AI-backed capability is added later, whether it follows OpenAI's pattern, Bedrock's, or a third provider is a decision to make deliberately each time, not by default.
- **New edge function duplicates only CORS/request-validation boilerplate from `parse-ai-command`**, not model-invocation code (the two functions' provider integrations are unrelated by design — see Decisions). Accepted for contract isolation.

## Migration Plan

No data migration — no new SQLite tables or schema version bump. Rollout is purely additive:
1. Ship `listPendingTodos()`/`countPendingTodos()` on the existing `todos` data layer (safe, additive, no behavior change to existing callers).
2. Provision the AWS Bedrock credentials/model-ID secrets (per Open Questions) in Supabase, then ship the `user-ai-ask` edge function (inert until called).
3. Ship the classify/retrieve/phrase orchestration and `AskConversationContext`, gated behind the same kind of feature flag already used for Create (`COMMAND_EXPERIMENT_ENABLED`-style gating), so it can ship dark and be enabled per the existing internal-rollout mechanism.
4. Add the Ask/Create/Auto mode toggle to the command center UI last, once the pipeline behind it is verified.

Rollback is deleting/disabling the new modules and edge function; nothing touches existing Create behavior or persisted data, so rollback has no data-cleanup step.

## Open Questions

- Whether `user-ai-ask` should log observation data (latency buckets, intent distribution) the same way `parse-ai-command` does today, for the same operability reasons.
- **AWS credentials mechanism for Bedrock — not yet decided.** Direct IAM access key/secret (simplest, but a long-lived credential to store as a Supabase secret) vs. an assumed role (`sts:AssumeRole`, short-lived credentials, more setup) — this is a deployment-time decision left open here, not asserted either way.
- **Exact Bedrock model ID for Claude Haiku — not yet decided.** A specific Bedrock model ID string (e.g. a `AI_ASK_BEDROCK_MODEL_ID`-style secret/config slot) is needed at deployment time; no specific ID is hardcoded into this design.
- **SigV4 request signing inside a Supabase Edge Function's Deno runtime — partially confirmed, not fully.** Spike result (task 4.2): a SigV4 signer built entirely from Web Crypto (`crypto.subtle.digest`/`crypto.subtle.sign` with HMAC-SHA256) — SHA-256 digest, HMAC-SHA256, the full `kDate → kRegion → kService → kSigning` key-derivation chain, and signature determinism — was implemented and cross-verified against Node's independent `node:crypto` module for the same inputs; all four checks matched exactly. Since Deno implements Web Crypto to the same W3C standard Node does, this is good evidence the cryptographic primitives a hand-rolled signer needs are portable and correct in Deno, with zero Node-specific/native dependency. **What this spike could NOT confirm**, because no `deno` binary or live Supabase/AWS access was available to test with: whether `@aws-sdk/client-bedrock-runtime` (or another AWS SDK v3 package) imports and runs cleanly inside the actual Supabase Edge Function Deno runtime, and whether a real signed request against live Bedrock succeeds end-to-end. The finding narrows the risk (a hand-rolled Web-Crypto-only signer is the lower-risk, verified-correct path) but does not fully close the open question — an actual Deno-runtime test against Supabase/Bedrock is still needed before this is considered settled.
